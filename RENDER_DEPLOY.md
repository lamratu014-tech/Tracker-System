# Deploying Ops & Planning to Render

This monorepo deploys to Render as three resources:

| Resource              | Render type | What it is                                        |
| --------------------- | ----------- | ------------------------------------------------- |
| `ops-planning-db`     | PostgreSQL  | App database (replaces the Replit Postgres)       |
| `ops-planning-api`    | Web Service | Express + Drizzle API (`artifacts/api-server`)    |
| `ops-planning-web`    | Static Site | Expo web SPA (`artifacts/mobile` exported to web) |

The included `render.yaml` (a Render Blueprint) creates all three in one go.

---

## 1. Push the code to GitHub

Render deploys from a Git repository, so the codebase needs to live on GitHub
(or GitLab/Bitbucket).

```bash
# from the project root
git remote -v                 # check if a remote already exists
git remote add origin git@github.com:<your-user>/<repo>.git   # if missing
git push -u origin main
```

If the repo already exists on GitHub but is out of date, just `git push`.

---

## 2. Create the Render Blueprint

1. Sign in at https://dashboard.render.com.
2. Click **New +** → **Blueprint**.
3. Connect the GitHub repo containing this code.
4. Render will detect `render.yaml` in the repo root and propose:
   - `ops-planning-db` (Postgres, free)
   - `ops-planning-api` (Web Service, free)
   - `ops-planning-web` (Static Site, free)
5. Click **Apply**.

Render will provision the database first, then build the API and the web SPA
in parallel. The first build will take 5–10 minutes.

> **Free-tier caveats** — Render's free Postgres expires after 90 days, and
> free web services sleep after 15 minutes of inactivity (cold start ≈ 30 s).
> Upgrade either to a paid plan for production.

---

## 3. Fill in the secrets Render couldn't auto-generate

In the Render dashboard, open **ops-planning-api → Environment** and set:

| Variable         | Value                                                                  |
| ---------------- | ---------------------------------------------------------------------- |
| `APP_DOMAIN`     | `https://ops-planning-web.onrender.com` (the SPA URL — no trailing /)  |
| `RESEND_API_KEY` | Your Resend API key (used for invite + reset emails)                   |

`DATABASE_URL`, `SESSION_SECRET`, and `SETUP_SECRET` are wired automatically
by `render.yaml` (the latter two are generated by Render).

Then open **ops-planning-web → Environment** and set:

| Variable                  | Value                                          |
| ------------------------- | ---------------------------------------------- |
| `EXPO_PUBLIC_API_BASE_URL` | `https://ops-planning-api.onrender.com`       |

This is baked into the SPA bundle at build time so the web app knows where to
reach the API. **After setting this, click "Manual Deploy → Deploy latest commit"**
on the static site so the new value is compiled in.

---

## 4. Push the database schema

The first time around, the `ops-planning-db` Postgres is empty. From your
local machine (or any host with Node + the repo cloned):

```bash
# Get the External Database URL from Render dashboard:
# ops-planning-db → Connect → External Database URL
export DATABASE_URL='postgresql://...render.com/...?sslmode=require'

pnpm install
pnpm --filter @workspace/db run push
```

This creates the tables. (Inside the API service container, the *Internal*
URL is used automatically — no action needed.)

---

## 5. Migrate data from the Replit Postgres (optional)

Skip this step if you want to start fresh.

```bash
# 1. Dump from Replit
export REPLIT_DB_URL='<your existing Replit Postgres URL>'
pg_dump --no-owner --no-acl --clean --if-exists \
  --format=custom \
  --file=replit-dump.pgcustom \
  "$REPLIT_DB_URL"

# 2. Restore into Render (use External URL)
export RENDER_DB_URL='<your Render Postgres External URL>'
pg_restore --no-owner --no-acl --clean --if-exists \
  --dbname="$RENDER_DB_URL" \
  replit-dump.pgcustom
```

If `pg_restore` complains about the `public` schema or extensions, re-run
with `--single-transaction` removed and let it skip already-present objects.

After restore, verify a few tables:

```bash
psql "$RENDER_DB_URL" -c 'SELECT count(*) FROM users;'
```

---

## 6. Bootstrap the first admin (only if you skipped data migration)

If the `users` table is empty:

```bash
curl -X POST https://ops-planning-api.onrender.com/api/auth/setup \
  -H 'Content-Type: application/json' \
  -d '{
        "email":"you@example.com",
        "name":"Your Name",
        "password":"a-strong-password",
        "department":"Ops",
        "setupSecret":"<value of SETUP_SECRET from Render dashboard>"
      }'
```

You can read `SETUP_SECRET` once from **ops-planning-api → Environment**.
After the first admin exists, this endpoint returns 409 forever.

---

## 7. Smoke test

Open the static site URL (`https://ops-planning-web.onrender.com`) in a
browser:

1. The login screen should render.
2. Sign in (or accept an invite).
3. Open DevTools → Network and confirm requests go to
   `https://ops-planning-api.onrender.com/api/...` and return 200.
4. Hit `https://ops-planning-api.onrender.com/api/healthz` directly — should
   return `200 OK`.

---

## 8. Decommission Replit (when you're satisfied)

1. Stop the Replit deployment from the Replit Deployments tab.
2. Keep the Repl around for at least a week as a rollback insurance policy.
3. Eventually delete the Replit Postgres (Render is now the source of truth).

## Troubleshooting

**Build fails with `ERR_PNPM_NO_LOCKFILE`** — make sure `pnpm-lock.yaml` is
committed to the repo and the Render service uses `pnpm install --frozen-lockfile`
(it already does in the bundled `render.yaml`).

**API logs show `self-signed certificate in certificate chain`** — that's
normal on Render's internal hostname; `lib/db/src/index.ts` already accepts
self-signed certs for known managed providers. If you point at a different
provider, set `PGSSL=require` to force the same behavior.

**Web app loads but every API call is CORS-blocked** — `EXPO_PUBLIC_API_BASE_URL`
isn't set on `ops-planning-web`, so the SPA is calling its own origin
instead of the API. Set the env var and redeploy the static site.

**Invite/reset emails point at `https://localhost`** — `APP_DOMAIN` isn't set
on `ops-planning-api`. Set it to the SPA URL and restart the service.

**Free Postgres about to expire** — Render shows a banner 14 days out.
`pg_dump` to back up, then either upgrade the existing DB or create a fresh
one and `pg_restore` into it.

## Web platform shims

The Expo app targets iOS, Android, and web from the same codebase. These
native-only modules need explicit handling so the web bundle (the Render
static site) doesn't crash:

- `expo-secure-store` — `lib/auth/storage.ts` lazy-requires it only when
  `Platform.OS !== "web"` and falls back to `window.localStorage` on web.
- `react-native-keyboard-controller` — `components/KeyboardAwareScrollViewCompat.tsx`
  swaps in a plain `ScrollView` on web; `KeyboardProvider` from
  `app/_layout.tsx` is a no-op on web and is safe to leave mounted.
- `expo-glass-effect` — `app/(tabs)/_layout.tsx` calls
  `isLiquidGlassAvailable()`, which the web build of the package returns
  `false` for, so the iOS-only `NativeTabs` path is never taken on web.
- `expo-symbols` — `SymbolView` is only rendered inside `Platform.OS === "ios"`
  branches; web/Android use `Feather` icons instead.
- `expo-haptics` — `components/MilestoneRow.tsx` guards `Haptics.impactAsync`
  with `Platform.OS !== "web"` because the package's web shim throws an
  `UnavailabilityError` when called.
- `expo-image-picker` and `expo-location` are listed as dependencies but
  are not currently imported anywhere in `app/`, so no shim is needed
  unless a future feature pulls them in.

To verify a clean web build locally, run
`pnpm --filter @workspace/mobile run build:web` and serve `artifacts/mobile/dist`
with any static server. The login screen should render and a `POST /api/auth/login`
round-trip should succeed against the Express API.

## CI guard: web build runs as part of `typecheck`

To prevent another native-only import (e.g. an unguarded `Haptics` call) from
silently breaking the Render web SPA, the root `pnpm run typecheck` script now
also runs the Expo web export:

```
pnpm run typecheck
  └─ tsc --build (libs)
  └─ per-artifact + scripts typecheck
  └─ pnpm run build:web
       └─ pnpm --filter @workspace/api-spec run codegen
       └─ pnpm --filter @workspace/mobile run build:web
```

Any error in the web bundle fails `typecheck`, which is what CI / the
pre-deploy script invokes. The same `build:web` script is what `render.yaml`
uses for the `ops-planning-web` static site, so a green `typecheck` locally
matches what Render will run.

**Skipping it locally** — if you only want a fast type pass during iteration
and don't care about the web bundle, run the steps individually instead of the
combined script:

```bash
pnpm run typecheck:libs
pnpm -r --filter "./artifacts/**" --filter "./scripts" --if-present run typecheck
```

Just remember to run the full `pnpm run typecheck` (or `pnpm run build:web`
on its own) before pushing — CI will run it for you regardless.
